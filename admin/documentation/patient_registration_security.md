## Overview

The patient registration system collects sensitive patient information and stores it in the database. Security is paramount to protect this data from unauthorized access, modification, or theft. The implementation includes multiple layers of security to address various potential vulnerabilities.

## Session Security

### Code Implementation
```php
session_start();
if (!isset($_SESSION['admin_id'])) {
    die("Access denied. Please login as admin.");
}
```

### Explanation
- `session_start()`: Initializes a session or resumes an existing one. Sessions allow data to be stored on the server and associated with a specific user through a session ID stored in a cookie.
- `$_SESSION`: A superglobal array that contains session variables. In this case, we check for 'admin_id' to verify the user is authenticated as an administrator.
- `die()`: Terminates script execution if the condition is met, preventing unauthorized access.

### Security Benefits
- **Authentication Verification**: Ensures only authenticated administrators can access the registration functionality.
- **Session Persistence**: Maintains user authentication state across multiple page requests.
- **Access Control**: Implements a basic authorization check before allowing access to sensitive functionality.

### Best Practices
- Sessions should timeout after a period of inactivity (typically 15-30 minutes).
- Session IDs should be regenerated after login to prevent session fixation attacks.
- Consider implementing IP binding for critical admin sessions.

## CSRF Protection

### Code Implementation
```php
// In register.php
if (!isset($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}
<input type="hidden" name="csrf_token" value="<?php echo $_SESSION['csrf_token']; ?>">

// In process-register-new.php
if (!isset($_POST['csrf_token']) || $_POST['csrf_token'] !== $_SESSION['csrf_token']) {
    echo "<div class='error'>Security validation failed. Please try again.</div>";
    exit;
}
```

### Explanation
- **CSRF (Cross-Site Request Forgery)**: An attack that tricks users into submitting forms or making requests they didn't intend to make.
- `bin2hex(random_bytes(32))`: Generates a cryptographically secure random token of 64 hexadecimal characters (32 bytes).
- The token is stored in the user's session and also included as a hidden field in the form.
- When the form is submitted, the token in the POST data is compared with the one in the session.

### Security Benefits
- **Request Verification**: Ensures that form submissions come from legitimate forms generated by your application.
- **Prevention of Forced Actions**: Stops attackers from forcing users to submit forms without their knowledge.
- **Session-Specific Protection**: Each user gets a unique token, making it impossible to create a generic attack.

### Best Practices
- Generate a new token after each form submission.
- Use a cryptographically secure random number generator.
- Make tokens sufficiently long (at least 128 bits/16 bytes).
- Consider implementing per-form or per-action tokens for critical operations.

## Input Sanitization

### Code Implementation
```php
function sanitize($input) {
    return htmlspecialchars(strip_tags(trim($input)));
}

$firstName = sanitize($_POST['first_name'] ?? '');
```

### Explanation
- `trim()`: Removes whitespace from the beginning and end of a string.
- `strip_tags()`: Removes HTML and PHP tags from a string.
- `htmlspecialchars()`: Converts special characters to HTML entities (e.g., < becomes &lt;).
- `??`: The null coalescing operator, returns the right operand if the left is NULL.

### Security Benefits
- **XSS Prevention**: Prevents Cross-Site Scripting attacks by neutralizing potentially malicious HTML/JavaScript.
- **Data Cleaning**: Removes unnecessary whitespace and ensures consistent data format.
- **Output Safety**: Makes data safe for display in HTML contexts.

### Best Practices
- Always sanitize user input before processing or storing it.
- Use context-specific sanitization based on where the data will be used.
- Consider using more specialized sanitization for specific data types (e.g., email addresses).

## Form Validation

### Code Implementation
```php
$errors = [];
if (empty($firstName)) $errors[] = "First name is required";
// Additional validation checks...

if (!filter_var($email, FILTER_VALIDATE_EMAIL)) $errors[] = "Invalid email format";
if (strlen($password) < 8) $errors[] = "Password must be at least 8 characters";

if (!empty($errors)) {
    // Display errors
}
```

### Explanation
- `empty()`: Checks if a variable is empty (null, 0, false, or empty string).
- `filter_var()`: Filters a variable with a specified filter. `FILTER_VALIDATE_EMAIL` validates email addresses.
- `strlen()`: Returns the length of a string.
- The validation results are collected in an array and displayed to the user if any errors are found.

### Security Benefits
- **Data Integrity**: Ensures required fields are provided and in the correct format.
- **Attack Surface Reduction**: Rejects malformed input before it reaches critical processing.
- **User Guidance**: Provides clear feedback to help users correct their input.

### Best Practices
- Validate both on the client-side (for user experience) and server-side (for security).
- Use specific validation for different data types (emails, dates, numbers, etc.).
- Consider implementing more complex validation rules for passwords (complexity requirements).

## Password Security

### Code Implementation
```php
$passwordHash = password_hash($password, PASSWORD_DEFAULT, ['cost' => 12]);
```

### Explanation
- `password_hash()`: Creates a password hash using a strong one-way hashing algorithm.
- `PASSWORD_DEFAULT`: Uses the bcrypt algorithm (currently), but will update to stronger algorithms as PHP evolves.
- `['cost' => 12]`: Sets the algorithmic cost factor, which determines how computationally intensive the hashing process is.

### Security Benefits
- **One-Way Hashing**: Passwords cannot be reversed from their hash, even if the database is compromised.
- **Salt Integration**: Automatically includes a random salt to prevent rainbow table attacks.
- **Future-Proofing**: The `PASSWORD_DEFAULT` constant will be updated to use stronger algorithms as they become available.
- **Computational Cost**: The cost factor makes brute-force attacks computationally expensive.

### Best Practices
- Never store plaintext passwords.
- Use `password_hash()` and `password_verify()` instead of older methods like md5 or sha1.
- Balance the cost factor between security and performance (higher is more secure but slower).
- Consider implementing additional password policies (complexity, rotation, etc.).

## SQL Injection Prevention

### Code Implementation
```php
$sql = "INSERT INTO patient (FirstName, MiddleName, Surname, Email, Contact, DateOfBirth, Gender, 
        EmergencyContact, Address, password, preferredDate, Notes) 
        VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)";

$stmt = $db_handle->prepare($sql);
$stmt->bind_param("ssssssssssss", 
    $firstName, 
    $middleName, 
    $lastName, 
    $email, 
    $phone, 
    $dateOfBirth, 
    $gender, 
    $emergencyContact,
    $address, 
    $passwordHash, 
    $preferredDate, 
    $notes
);
```

### Explanation
- **SQL Injection**: An attack where malicious SQL code is inserted into queries through user input.
- `prepare()`: Creates a prepared statement template.
- `?`: Placeholder for parameters that will be bound later.
- `bind_param()`: Binds variables to the parameter placeholders.
  - The first argument ("ssssssssssss") specifies the data types of each parameter:
    - "s" for string
    - "i" for integer
    - "d" for double
    - "b" for blob

### Security Benefits
- **Query/Data Separation**: The SQL structure is sent to the database separately from the data.
- **Automatic Escaping**: The database driver properly escapes all special characters.
- **Type Safety**: Data is treated according to its specified type.
- **Prevention of SQL Syntax Manipulation**: User input cannot change the structure of the query.

### Best Practices
- Always use prepared statements for queries with user input.
- Specify the correct data types for each parameter.
- Consider using an ORM (Object-Relational Mapping) library for more complex applications.
- Implement the principle of least privilege for database users.

## Database Connection Security

### Code Implementation
```php
$db_handle = getDatabaseConnection();
```

### Explanation
- `getDatabaseConnection()`: A function defined in `../../config/database.php` that establishes a connection to the database.
- This centralizes database credentials and connection logic.

### Security Benefits
- **Credential Management**: Database credentials are stored in a single location, not scattered throughout the codebase.
- **Connection Standardization**: Ensures all database connections use the same security settings.
- **Reduced Exposure**: Credentials are not hardcoded in publicly accessible files.

### Best Practices
- Store database credentials outside the web root if possible.
- Use environment variables for sensitive configuration in production.
- Implement connection pooling for performance in high-traffic applications.
- Use a database user with minimal required privileges.

## Error Handling

### Code Implementation
```php
if ($stmt->execute()) {
    $patientId = $db_handle->insert_id;
    echo "<div class='success'>Patient record created successfully! ID: $patientId</div>";
    error_log("Patient registration successful for email: $email, ID: $patientId");
} else {
    echo "<div class='error'>Error: Registration failed. Please try again.</div>";
    error_log("Patient registration failed: " . $stmt->error);
}
```

### Explanation
- `$stmt->execute()`: Executes the prepared statement and returns a boolean indicating success or failure.
- `$db_handle->insert_id`: Returns the ID generated by the last INSERT operation.
- `error_log()`: Sends an error message to the web server's error log.
- Generic error messages are shown to users, while detailed errors are logged.

### Security Benefits
- **Information Disclosure Prevention**: Detailed error information is not exposed to users.
- **Audit Trail**: Errors are logged for troubleshooting and security monitoring.
- **User Experience**: Users receive appropriate feedback without technical details.

### Best Practices
- Never display database errors, stack traces, or system paths to users.
- Log detailed error information for administrative review.
- Implement structured logging for easier analysis.
- Consider implementing a monitoring system for critical errors.

## Logging and Audit Trail

### Code Implementation
```php
error_log("Admin {$_SESSION['admin_id']} attempting to register patient: $email");
error_log("Patient registration successful for email: $email, ID: $patientId");
```

### Explanation
- `error_log()`: Sends a message to the system's error log or a specified file.
- The logs include the admin ID, action being performed, and relevant identifiers.

### Security Benefits
- **Accountability**: Actions are traced to specific admin users.
- **Intrusion Detection**: Unusual patterns of activity can be identified.
- **Forensic Analysis**: In case of a security incident, logs provide evidence of what happened.
- **Compliance**: Many regulatory frameworks require audit logging.

### Best Practices
- Log all security-relevant events (login attempts, data modifications, etc.).
- Include timestamps, user identifiers, and action details in logs.
- Protect log files from unauthorized access.
- Consider implementing a centralized logging system for larger applications.

## Glossary of PHP Functions

### Session Functions
- `session_start()`: Starts a new session or resumes an existing one.
- `session_regenerate_id()`: Generates a new session ID.
- `session_destroy()`: Destroys all data registered to a session.

### String Functions
- `trim()`: Removes whitespace from the beginning and end of a string.
- `strip_tags()`: Removes HTML and PHP tags from a string.
- `htmlspecialchars()`: Converts special characters to HTML entities.
- `strlen()`: Returns the length of a string.
- `bin2hex()`: Converts binary data to hexadecimal representation.

### Array Functions
- `empty()`: Checks if a variable is empty.
- `isset()`: Determines if a variable is set and is not NULL.
- `array()`: Creates an array.
- `[]`: Array syntax for adding elements.

### Database Functions (MySQLi)
- `mysqli_connect()`: Opens a new connection to the MySQL server.
- `prepare()`: Prepares an SQL statement for execution.
- `bind_param()`: Binds variables to a prepared statement as parameters.
- `execute()`: Executes a prepared statement.
- `fetch()`: Fetches the result from a prepared statement into bound variables.
- `close()`: Closes a prepared statement or database connection.
- `insert_id`: Returns the auto-generated ID used in the last query.

### Cryptography Functions
- `password_hash()`: Creates a password hash.
- `password_verify()`: Verifies that a password matches a hash.
- `random_bytes()`: Generates cryptographically secure pseudo-random bytes.

### Validation Functions
- `filter_var()`: Filters a variable with a specified filter.
- `FILTER_VALIDATE_EMAIL`: Validation filter for email addresses.

### Error Handling Functions
- `error_log()`: Sends an error message to the defined error handling routines.
- `die()`: Equivalent to exit(), terminates script execution.
- `exit()`: Terminates script execution.
